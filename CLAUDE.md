## Build and Test Commands

```bash
# Build (from repo root)
dotnet build Expensify.slnx -v minimal

# Build without NSwag client generation (faster)
dotnet build Expensify.slnx -v minimal -p:NoSwagGen=true

# Run all tests
dotnet test Expensify.slnx -v minimal

# Run a single test project
dotnet test tests/Expensify.Modules.Users.UnitTests -v minimal

# Run a single test by name
dotnet test Expensify.slnx --filter "FullyQualifiedName~RegisterUserCommandHandlerTests"

# Run the API locally (auto-applies migrations and seeds in Development)
cd src/API/Expensify.Api && dotnet run

# Start infrastructure (PostgreSQL, Redis, Aspire Dashboard)
docker compose up -d
```

## Architecture

Modular monolith on .NET 10 / C# 14. Clean Architecture boundaries enforced by NetArchTest.

- **Host**: `src/API/Expensify.Api` (composition root: `Program.cs`)
- **Common**: `src/Common/*` (shared Domain, Application, Infrastructure, Presentation)
- **Modules**: `src/Modules/*` with layers: Domain, Application, Infrastructure, Presentation, optional IntegrationEvents
- **Tests**: `tests/` (unit, architecture, integration)

### Layer dependency rules (enforced by architecture tests)
- Domain must not reference Application, Infrastructure, or Presentation
- Application must not reference Infrastructure or Presentation
- Presentation must not reference Infrastructure

### Request flow
1. Carter endpoint (Presentation) maps HTTP input, sends MediatR command/query
2. Application handler (`internal sealed`) executes use case, returns `Result<T>`
3. Infrastructure persists via DbContext/repositories + `IUnitOfWork`
4. Domain events captured by `InsertOutboxMessagesInterceptor`, dispatched by Quartz `ProcessOutboxJob`

### Key patterns
- **CQRS**: `ICommand<T>`/`IQuery<T>` with dedicated handlers; pipeline behaviors for validation, logging, exception handling
- **Result pattern**: `Result`/`Result<T>` instead of exceptions; strongly-typed `Error`/`ValidationError`
- **Outbox/Inbox**: domain events persisted transactionally, dispatched asynchronously; inbox deduplicates integration events
- **Module registration**: each module exposes `Add<Name>Module(IServiceCollection, IConfiguration)` in Infrastructure

## Hard Conventions

- Warnings are errors (`TreatWarningsAsErrors` in `Directory.Build.props`)
- NuGet versions centralized in `Directory.Packages.props`
- Application handlers/validators must be `internal sealed` with naming conventions (`*Command`, `*CommandHandler`, `*Query`, `*QueryHandler`, `*Validator`, `*DomainEventHandler`)
- Domain events must be `sealed` and end with `DomainEvent`; entities need private parameterless constructor
- Do not manually edit `src/API/Expensify.Api.Client/ExpensifyV1Client.g.cs`
- JWT user identity is source of truth for user-scoped operations; enforce authorization policies per endpoint
- Nullable reference types enabled globally

## Adding a New Module

Mirror the Users module structure. Register in `Program.cs`: add Application assembly to `moduleApplicationAssemblies`, call `Add<Name>Module`, add to `AddModuleConfiguration`, add migration call in `MigrationExtensions.ApplyMigrations`. Add architecture tests under `tests/`.

## Integration Tests

Integration tests use Testcontainers (PostgreSQL, Redis). `ApiDriver.cs` overrides config and replaces external services (e.g., SendGrid) with in-memory test doubles.
